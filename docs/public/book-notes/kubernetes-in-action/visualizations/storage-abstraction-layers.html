Pod（开发者视角）
只需引用 PVC 名称，无需了解底层存储
Namespace 级
Pod 的 volumes 字段引用 PVC 名称；容器通过 volumeMounts 挂载。整个过程对底层存储技术完全透明——同一份 YAML 可以部署到任意云平台。
引用方式

AI 场景
推理服务挂载模型权重 (ReadOnlyMany)

📋
PersistentVolumeClaim（PVC）
用户的存储申请单：声明容量、访问模式
Namespace 级
PVC 是开发者创建的存储"申请单"。它不指定任何基础设施细节，只说明 需要多少存储 和 什么访问模式。Kubernetes 自动寻找满足条件的 PV 进行绑定。
关键字段
storage: 1Gi, accessModes: RWO
绑定触发
创建后自动由 Kubernetes 完成绑定
storageClassName: ""
强制绑定预创建的 PV（禁用动态供应）
storageClassName: fast
触发 StorageClass 动态供应新 PV
自动绑定 ↓
💾
PersistentVolume（PV）
集群存储池：管理员创建或 StorageClass 动态生成
集群级
PV 是集群管理员预置的存储资源，描述实际底层存储的容量、访问模式和回收策略。PV 不属于任何 Namespace，是集群级别的资源。
生命周期状态
Available → Bound → Released
回收策略
Retain（保留）/ Delete（删除底层存储）
StorageClass 动态供应 或 管理员手动创建 ↓
⚙️
StorageClass
存储"配方"：定义 provisioner 和参数，自动创建 PV
集群级
StorageClass 让管理员无需预创建大量 PV，而是定义"如何创建存储"的规则。PVC 引用 StorageClass 后，provisioner 自动创建符合要求的 PV 并完成绑定。
provisioner 示例
kubernetes.io/gce-pd, k8s.io/aws-ebs
默认 StorageClass
PVC 不指定 storageClassName 时自动使用
底层实现 ↓
🗄️
实际存储基础设施
GCE PD / AWS EBS / NFS / Ceph — 对开发者透明
基础设施层
真实的物理或网络存储系统。由云提供商或 IT 团队维护，通过 StorageClass provisioner 暴露给 Kubernetes。开发者完全无需感知这一层的存在。
云存储
GCE PD, AWS EBS, Azure Disk
网络存储 (NFS/Ceph)
支持 ReadWriteMany，适合分布式训练
AI 推理模型存储
NFS ReadOnlyMany，多推理 Pod 并发挂载
训练 Checkpoint
Ceph RBD / NFS ReadWriteMany

1
Admin 部署 StorageClass
集群管理员创建 StorageClass 资源（如 fast），指定 provisioner 为 kubernetes.io/gce-pd，type 为 pd-ssd。也可将某个 StorageClass 标记为默认。
2
开发者创建 PVC（引用 StorageClass）
开发者提交 PVC manifest，指定 storageClassName: fast、所需容量和访问模式。此时无任何 PV 存在，PVC 处于 Pending 状态。
3
Kubernetes 查找 StorageClass
Kubernetes 控制平面检测到新 PVC，查找其引用的 StorageClass fast，读取其 provisioner 配置和参数。
4
Provisioner 自动创建实际存储
Provisioner（kubernetes.io/gce-pd）被调用，在 GCP 上自动创建一块 SSD 磁盘，容量和 zone 按 StorageClass parameters 和 PVC 请求确定。
5
自动创建 PV 并绑定到 PVC
Provisioner 创建 PV 资源，引用刚才创建的磁盘，并立即将 PV 与 PVC 绑定。PVC 状态变为 Bound。动态供应的 PV 默认 reclaimPolicy 为 Delete。
6
Pod 创建并挂载存储
开发者创建 Pod，volumes 中引用 PVC 名称。Kubernetes 调度器将 Pod 调度到节点后，kubelet 自动将 PV 对应的磁盘挂载到容器指定路径。全程无需人工干预。
