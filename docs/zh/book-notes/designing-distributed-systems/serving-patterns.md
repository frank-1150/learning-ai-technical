# 服务模式：负载均衡、分片、Scatter/Gather 与事件驱动

> 本文是 Brendan Burns 所著 *Designing Distributed Systems, 2nd Edition* 第三部分（Chapter 6-10）的阅读笔记。这一部分介绍了五种多节点服务模式（Serving Patterns），从最简单的负载均衡复制到复杂的分布式所有权选举，覆盖了构建生产级分布式系统最核心的架构选择。

第一部分和第二部分分别讨论了单节点模式（Sidecar、Ambassador、Adapter）和多容器协作模式。第三部分则将视角拉到了**多机协作**的层面：当你的系统需要跨越多台机器来服务用户时，应该选择哪种架构模式？

## 从单体到微服务：模式的必要性

在进入具体模式之前，书中花了一些篇幅讨论微服务架构的利与弊。微服务将大型应用拆分为多个独立组件，通过 API 通信。其优势包括：

- **可靠性和敏捷性**：每个服务由一个小团队维护（"两个披萨"团队），独立发布
- **解耦**：API 作为正式契约，降低了团队间的同步成本
- **独立扩展**：不同服务可以选择最适合自身的扩展策略

但微服务也带来了挑战——调试困难、架构设计复杂、通信开销增加。Burns 给出了一个实用的经验法则：**如果微服务的数量超过了工程师的数量，你可能过度拆分了**。

正是这些挑战催生了对分布式设计模式的需求。如果微服务架构由一组知名模式组成，那么设计和调试都会变得更加可控。

## Ch6 复制负载均衡服务（Replicated Load-Balanced Services）

这是最简单也最常见的分布式模式：每个服务器完全相同，前面放一个负载均衡器。

### 无状态服务

无状态服务不需要保存状态，每个请求可以路由到任意副本。典型例子包括静态内容服务器和中间件聚合层。为了实现高可用（三个九 = 99.9%），至少需要两个副本——单实例在滚动更新或崩溃时无法保证 SLA。

在 Kubernetes 中，这通过 `Deployment` + `Service` 实现：Deployment 定义副本数和容器镜像，Service 提供负载均衡和 DNS 名称。

### Readiness Probe：不只是活着

一个关键细节是 **就绪探针（Readiness Probe）**。容器启动后可能需要连接数据库、加载数据，此时它虽然"活着"但并不"就绪"。就绪探针告诉负载均衡器何时可以开始接收流量，避免将请求发送到尚未初始化完成的实例。

### 会话追踪服务

纯轮询的负载均衡不总是最优的。有时候需要将同一用户的请求路由到同一副本（例如利用本地缓存提高命中率，或维护长连接状态）。这通过 **会话追踪（Session Tracking）** 实现，通常基于源 IP 哈希或 Cookie。

> 需要注意的是，基于 IP 的会话追踪在 NAT 环境下效果不佳，因为多个用户可能共享同一个外部 IP。对于外部流量，基于 Cookie 或 HTTP Header 的应用层追踪更可靠。

### 缓存层

当无状态服务的计算成本较高（如数据库查询、渲染），在前面加一层缓存代理（如 Varnish）可以显著降低延迟。缓存部署有两种方式：

| 部署方式 | 优点 | 缺点 |
|---------|------|------|
| **Sidecar**（与应用同 Pod） | 简单，无额外网络跳转 | 缓存必须与应用同等扩展，浪费内存 |
| **独立复制层** | 缓存可独立扩展，少量大内存实例效率更高 | 增加架构复杂度 |

书中推荐使用独立复制层：用少量大内存的缓存实例，搭配多个小的应用副本。这样缓存的内存利用率更高（避免每个副本存储完全相同的数据）。

### 限流与 SSL 终结

缓存层不仅仅用于缓存，还可以附加其他功能：

- **限流**：通过 HTTP 反向代理的 throttle 模块，基于 IP、路径或用户身份进行速率限制。API 通常返回 `429` 状态码和 `X-RateLimit-Remaining` 头
- **SSL 终结**：在最外层部署 nginx 处理 HTTPS，将解密后的 HTTP 流量转发给内部缓存和应用层

最终的完整架构是三层复制：**nginx（SSL）-> Varnish（缓存）-> 应用服务器**，每一层都可以独立扩展。

## Ch7 分片服务（Sharded Services）

复制服务中每个副本是同质的，而分片服务中每个副本（shard）只负责请求的一个子集。负载均衡节点（root）检查每个请求并将其路由到对应的分片。

### 为什么需要分片？

主要原因是**数据规模超出了单机容量**。以缓存为例：10 个副本各 10GB RAM，如果使用复制模式，每个副本存储相同数据，实际只能缓存总数据集的 5%（10GB/200GB）。而使用分片，10 个分片各存储不同数据，可以缓存 50%（100GB/200GB），命中率提升 10 倍。

分片还有一个额外好处——**故障隔离**。在复制系统中，一个"毒请求"（poison request）可能击溃所有副本；而在分片系统中，毒请求只影响对应的那个分片。

### 分片函数的设计

分片的核心是 **分片函数**：`Shard = hash(Key) % N`。设计分片函数需要关注：

- **确定性**：相同输入始终映射到同一分片
- **均匀性**：输出均匀分布在所有分片上
- **分片键的选择**：键过于具体（如整个请求对象）会导致不必要的分散；过于宽泛则会合并不该合并的请求。例如对 HTTP 缓存，`shard(country(request.ip), request.path)` 比简单的 `shard(request)` 更合理

### 一致性哈希

当分片数从 N 变为 N+1 时，简单取模（`hash(key) % N` -> `hash(key) % (N+1)`）会导致**大量键重新映射**，相当于缓存全部失效。

**一致性哈希（Consistent Hashing）** 解决了这个问题：扩容时只需要重映射约 `K/N` 个键（K 是总键数，N 是新的分片数），而不是几乎所有键。例如从 10 个分片扩展到 11 个，只有约 10% 的键需要迁移。

### 复制分片服务与热分片

对于不能容忍分片故障的场景，可以将每个分片本身部署为一个**复制服务**（replicated sharded service），这样即使某个分片的一个副本宕机，该分片仍然可用。

**热分片（Hot Sharding）** 处理的是负载不均的问题。当某个分片流量突然激增（例如一张病毒传播的照片命中了同一个缓存分片），系统可以动态地将该分片复制到更多机器上，同时将负载较低的分片合并到更少的机器。

## Ch8 Scatter/Gather 模式

前面的模式处理的是"扩展请求量"（复制）和"扩展数据量"（分片），而 Scatter/Gather 处理的是**扩展计算时间**——通过并行化来降低单个请求的延迟。

### 工作原理

Root 节点收到请求后，将其同时发送（scatter）给所有叶子节点。每个叶子处理请求的一部分，将结果返回给 Root。Root 汇总（gather）所有结果后返回给用户。

两种变体：

| 变体 | 叶子节点特征 | 适用场景 |
|------|-----------|---------|
| **Root Distribution** | 同质，每个叶子处理完整数据集的相同计算的不同部分 | 计算密集型任务（如大规模并行计算） |
| **Leaf Sharding** | 异质，每个叶子持有数据集的不同子集 | 数据密集型搜索（如分布式文档索引） |

书中举了一个分布式文档搜索的例子：搜索"cat AND dog"时，Root 将查询 scatter 到所有叶子节点，每个叶子返回匹配的文档列表，Root 计算交集后返回最终结果。

### 尾延迟问题（Straggler Problem）

Scatter/Gather 有一个关键缺陷：**总延迟取决于最慢的叶子节点**。

假设单个请求的 P99 延迟为 2 秒（即 1% 的概率耗时 2 秒）。如果你 scatter 到 5 个叶子，至少有一个慢响应的概率是 `1 - 0.99^5 = 4.9%`。Scatter 到 100 个叶子？概率接近 100%——你的**系统级 P99 基本上就是 2 秒**。

这带来三个重要启示：

1. 增加并行度并不总是加速——每个叶子的开销是固定的，当开销超过有效计算时，加节点反而变慢
2. **P99 比 P50 重要得多**——因为每个用户请求实际上是多个子请求
3. 必须通过**复制每个分片**来提高可靠性，避免单点故障

### 扩展可靠性

正确的做法是将每个叶子分片实现为一个复制服务。这样每个叶子请求实际上由负载均衡器分发到健康的副本，确保单机故障不会影响用户，也允许安全地进行滚动升级。

## Ch9 函数与事件驱动处理（FaaS）

前面的模式都是**长驻服务**（always-on）。但有些场景只需要临时创建来处理单个请求或事件——这就是 FaaS（Function as a Service）/ 事件驱动架构。

> FaaS 常被称为 "Serverless"，但两者并不完全等同。多租户容器编排平台是 serverless 但不是事件驱动的；自建集群上的 FaaS 是事件驱动但不是 serverless 的。

### FaaS 的优势与挑战

| 维度 | 优势 | 挑战 |
|------|------|------|
| **开发体验** | 从代码到运行极其简单，自动扩展 | 函数间依赖难以追踪，调试困难 |
| **架构** | 天然解耦，强制模块化 | 无本地状态，所有状态需外部存储 |
| **成本** | 低流量时按请求付费极划算 | 高流量时按请求定价反而比长驻实例贵 |
| **性能** | 自动扩缩容 | 冷启动延迟；需大量内存数据的服务不适合 |

一个经典的陷阱：如果 `functionA()` 调用 `functionB()` 调用 `functionC()` 再调用 `functionA()`，就形成了无限循环——而且在 FaaS 环境中这种循环依赖**极难检测**，因为每个函数是完全独立部署的。

### Decorator 模式

FaaS 非常适合实现**请求/响应转换**。Decorator Function 拦截请求、修改后再转发给主服务。典型用例：

- 为 API 请求添加默认值（如果 `name` 字段为空则随机生成）
- 请求校验和鉴权
- 响应格式转换

这种模式的好处是转换逻辑与主服务**独立扩展**。

### 事件处理与事件管道

FaaS 的最佳应用场景是**事件处理**。事件是单次触发的异步操作（用户注册发送欢迎邮件、文件上传触发通知），与请求-响应的会话模式不同。

**事件管道（Event Pipeline）** 将多个函数串联成有向图。例如代码提交的 CI/CD 管道：

```
代码提交 -> 触发构建 -> 构建分析函数
  |-> 构建成功 -> 创建审批工单 -> 工单关闭 -> 推送生产
  |-> 构建失败 -> 创建 Bug 报告 -> 管道终止
```

书中还展示了一个用户注册管道的实现：主函数维护一个必需操作列表（发送欢迎邮件）和一个可选操作列表（订阅邮件通知），每个操作都是独立的 FaaS，通过 webhook 调用串联。这种设计让添加新的注册后操作变得极其简单——只需要添加一个新函数和对应的 webhook。

## Ch10 所有权选举（Ownership Election）

最后一个模式解决的是**分布式所有权**问题：当多个副本存在时，如何确保只有一个副本拥有某个资源或任务的控制权？

### 需不需要 Leader Election？

最简单的方法是运行单实例（Singleton）。在 Kubernetes 中，单实例有以下保障：

- 容器崩溃 -> 自动重启（秒级）
- 容器 hang -> 健康检查失败后重启
- 机器故障 -> 迁移到其他节点（分钟级）

如果每天崩溃一次，大约能达到三到四个九的可用性（99.95%+）。对于后台异步处理等场景，这可能就够了。但对于交互式系统（游戏、银行、零售），即使短暂的中断也可能造成严重的客户不满，这时就需要真正的 Leader Election。

### 基于分布式键值存储的实现

实现 Leader Election 有两种路径：直接实现共识算法（Paxos/Raft），或基于已有的分布式键值存储（etcd、ZooKeeper、Consul）。书中推荐后者，因为这些系统已经实现了共识并提供了两个关键原语：

- **Compare-and-Swap（CAS）**：只有当前值匹配预期值时才写入新值
- **TTL（Time-to-Live）**：键到期自动删除

基于这两个原语可以构建：

**分布式锁**：通过 CAS 写入锁的值（"1" 表示锁定，"0" 表示解锁），加上 TTL 防止持有者崩溃后锁永远不被释放。

**可续约租约（Renewable Lease）**：锁持有者定期（每 TTL/2）续约，确保在正常运行时不会意外过期。如果续约失败，系统调用 `handleLockLost()` 终止当前工作，让编排系统重启该副本。

### 并发数据操作的陷阱

即使有了锁，仍然可能出现**两个副本同时认为自己是 leader** 的短暂窗口。考虑以下场景：

1. Process-1 获取锁（TTL=t）
2. Process-1 运行过慢，超过 t
3. 锁过期，Process-2 获取锁
4. Process-1 完成后调用 unlock —— **实际上解锁了 Process-2 的锁**
5. Process-3 趁机获取锁

解决方案是使用 **资源版本（Resource Version）**：每次写入都会更新版本号，unlock 时必须验证版本号匹配。这样 Process-1 的 unlock 会因版本不匹配而失败，不会影响 Process-2。

书中最终指出了一个根本性的权衡：

- **最多一次处理（at-most-once）**：宁可短暂不可用，也不重复处理。适用于金融交易
- **至少一次处理（at-least-once）**：宁可偶尔重复处理，也不丢失。适用于可幂等的计算任务

## 五种模式的总览

| 模式 | 扩展维度 | 典型场景 | 关键挑战 |
|------|---------|---------|---------|
| **复制负载均衡** | 请求吞吐量 | 无状态 Web 服务 | 缓存一致性、会话追踪 |
| **分片服务** | 数据规模 | 缓存、游戏世界 | 分片键设计、一致性哈希 |
| **Scatter/Gather** | 计算延迟 | 分布式搜索 | 尾延迟、叶子数量权衡 |
| **FaaS/事件驱动** | 开发速度 | 事件处理、API 增强 | 调试、冷启动、循环依赖 |
| **所有权选举** | 排他控制 | 调度器、分布式锁 | 并发安全、at-most/at-least-once |

这五种模式并不互斥——一个生产级分布式系统通常是多种模式的组合。例如一个搜索引擎可能使用 Scatter/Gather 处理查询，每个叶子分片使用复制服务保证可靠性，FaaS 处理索引更新事件，Leader Election 管理索引构建任务的所有权。理解每种模式的适用场景和权衡，是设计可靠分布式系统的基础。
