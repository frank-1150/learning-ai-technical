# 通用概念：可观测性、AI 推理服务与常见故障模式

《Designing Distributed Systems》前四部分介绍了从单节点到复杂分布式系统的各种构建模式。Part V 则跳出具体模式，聚焦于**所有分布式系统都需要面对的通用概念**：如何观测系统运行状态、如何集成 AI 推理能力、以及如何避免那些反复出现的经典故障。这三章内容看似独立，实则构成了分布式系统"运营三角"——你不仅要会建系统，更要会看系统、会用 AI 增强系统、会从别人的失败中学习。

## 第 14 章：监控与可观测性模式

分布式系统与客户端应用的核心区别在于：服务是 7x24 小时运行的，面向全球用户。这意味着**你必须在用户之前发现问题**，而发现问题的前提是能够"看见"系统的运行状态。

### 可观测性四大支柱

监控与可观测性解决方案由四个核心概念组成：

| 支柱 | 关注点 | 典型工具 |
|------|--------|----------|
| **日志 (Logging)** | 记录代码中特定位置的执行信息，帮助理解单次请求的行为 | ELK Stack, Loki |
| **指标 (Metrics)** | 收集跨请求的聚合统计数据，反映服务整体运行状态 | Prometheus, Grafana |
| **告警 (Alerting)** | 基于日志或指标的规则，主动通知团队关注问题 | PagerDuty, Alertmanager |
| **追踪 (Tracing)** | 将分散在多个微服务中的请求串联起来，还原端到端视角 | OpenTelemetry, Jaeger |

### 日志：从 printf 到生产级实践

日志是最古老也最直觉的可观测性手段。但在分布式系统中，"log spam" 是个真实的问题——当一切正常时，大部分日志都是噪音。

**分级日志**是应对之道。常见的日志级别包括：

- **Debug**：仅在调试时有用的详细信息
- **Info**：有用但非异常的操作信息
- **Warning**：令人担忧但不致命的情况
- **Error**：需要立即关注的严重问题
- **Fatal**：不可恢复的错误，应用将在写完日志后终止

一个实用的经验法则：**记录你在事后复盘时希望拥有的信息**。书中提到一个程序员之间的经典笑话——知道该记录什么的最好方式，就是回忆上次调试时你缺少什么信息。

**条件日志 (Conditional Logging)** 也值得注意。你可以针对特定用户或特定组件动态开启 debug 级别日志，而不影响其他请求的性能。但要小心：在排查故障期间开启条件日志本身不应该引发更严重的故障。

### 指标：Prometheus 与时间序列

与日志关注个体执行不同，指标收集的是跨时间的聚合统计。Prometheus 已成为事实标准，它支持三种基本数据类型：

- **直方图 (Histogram)**：值的分布，如延迟的 P50/P90/P99
- **计数器 (Counter)**：只增不减的累积值，如总请求数
- **仪表 (Value/Gauge)**：可增可减的瞬时值，如 CPU 使用率、内存占用

Prometheus 是拉取 (pull-based) 模型——定期向应用发送请求获取指标。对于批处理等短生命周期的任务，可以使用 **Prometheus Push Gateway** 实现推送模式。两者结合，才能获得完整的运行视图。

### 请求监控：从基础到进阶

**基础请求监控**围绕三个核心指标：

1. **请求速率**：总请求数的变化率，注意选择合适的时间窗口来平滑毛刺
2. **响应码分布**：按 HTTP 状态码（2xx/4xx/5xx）细分，比总数更有价值
3. **延迟直方图**：不仅看平均值，更要看 P99——那是你最慢用户的体验

**进阶请求监控**则深入更多维度：

- **请求/响应体大小**：可以关联到慢请求的原因
- **队列时间 vs 处理时间**：区分"需要扩容"还是"需要优化代码"
- **地理/客户维度**：将指标按地域或客户拆分，发现局部问题

书中的一个关键洞察：**监控和日志是互补的**。指标提供快速的聚合视图（"P99 延迟飙高了"），日志则提供定位根因所需的细节（"到底是哪个函数慢了"）。

### 告警：SLO 驱动的设计

告警的核心思路是定义**服务级别目标 (SLO)**——比如"P90 延迟低于 500ms"或"5xx 错误率低于 1%"。当指标偏离 SLO 时触发告警。

设计告警时需要平衡两个维度：

- **过于灵敏**：频繁误报导致"狼来了"效应，值班工程师逐渐忽略告警
- **过于迟钝**：漏掉真实问题，最终变成用户报告事故 (CRI)

书中还提到了**异常检测告警**——当静态阈值不够用时，AI 驱动的异常检测可以发现"客户 A 完全不可用但整体指标正常"这类问题。随着系统规模增长，这将从"有则更好"变为"必须具备"。

### 追踪：OpenTelemetry 的端到端视角

在微服务架构中，一个用户请求可能经过十几个服务。追踪的核心思想是：**为每个请求生成唯一的 correlation ID**，在所有服务间传递。这样就能将分散的日志和指标关联起来，还原完整的请求路径。

具体实现上，可以通过 HTTP 头（如 `x-my-apps-correlation-id`）传递 ID。但自己实现这套体系工作量巨大，书中强烈推荐使用 **OpenTelemetry**——它提供多语言 SDK、支持多种后端、且有丰富的可视化生态。

### 信息聚合与降采样

监控产生的数据量惊人，需要合理的聚合策略：

- **跨进程聚合**：使用 `ktail` 或 Elasticsearch 将多个副本的日志汇总
- **时间维度降采样**：比如一周后将每秒一次的指标降为每分钟一次
- **冷热存储分层**：高频访问数据放"热"存储，历史数据下沉到"冷"存储（如对象存储）

## 第 15 章：AI 推理与服务

AI 已成为分布式系统中的重要组件。这一章从分布式系统工程师的视角介绍如何将 AI 集成到应用中。

### AI 系统基础

- **模型**是一组数值权重的集合，现代 LLM 有数万亿参数
- **训练**：用大量输入-输出对训练模型，计算密集且需要专用硬件
- **微调 (Fine-tuning)**：在预训练模型基础上用较小数据集精调，更经济
- **推理 (Inference)**：将模型应用于用户输入，通常以 RESTful API 形式提供
- **Prompt 工程**：构造合适的提示词以引导模型产出高质量输出

### 模型托管

托管模型时需要考虑的关键因素：

- **延迟敏感**：用户在 250ms 开始感知延迟，几秒后就会转移注意力
- **GPU 需求**：LLM 通常需要高性能 GPU，成本高且难以获取
- **小语言模型 (SLM)**：如 Microsoft Phi 系列，用更少参数实现可接受的质量，甚至能在移动端运行，同时兼顾隐私

### 模型分发

- 推荐使用 **ONNX** 格式——跨框架的开放标准
- **Hugging Face** 是最流行的模型中心，Python 一行代码即可开始使用
- **缓存模型**到本地文件系统或内存 KV 存储，避免每次启动都重新下载
- **安全部署**：遵循渐进式发布的最佳实践——先对一小部分用户灰度，逐步扩大

### 开发与成本优化

推理成本不菲，因此开发阶段的一个实用策略是：**通过 RESTful API 抽象模型选择**。开发和测试环境使用廉价的小模型，生产环境切换为完整模型。可以用 Kubernetes Service 资源来管理这种切换。

### RAG：检索增强生成

RAG 解决了 LLM 的两个固有限制：**知识固定于训练时间点**和**缺乏用户个人上下文**。

核心思路是在用户 prompt 中注入从传统数据库检索到的实时信息。例如用户问"离我最近的朋友是谁？"，系统会先查询联系人列表和位置服务，然后将结果注入 prompt。RAG 的额外优势是**保护隐私**——用户数据不会编码进模型本身。

### AI 应用的测试范式转变

传统测试验证 `Value(A).equals(B)`，但 AI 应用需要从**正确性测试转向统计质量测试**：

- 单一输入的输出可能因模型变化而剧烈波动
- 需要在大量 prompt 样本上评估**整体质量是否提升**
- 可以用 LLM 自身来评估响应质量（"LLM-as-Judge"模式）
- 上线后要持续监控用户反馈信号，而不仅仅看错误率和延迟

## 第 16 章：常见故障模式

这是全书最特别的一章——不是教你怎么建，而是教你**怎么不建**。这些故障模式是无数次线上事故的经验结晶。

### 故障模式总览

| 故障模式 | 核心问题 | 缓解策略 |
|----------|----------|----------|
| **惊群效应 (Thundering Herd)** | 失败请求重试导致流量指数级放大 | 指数退避 + Jitter + 熔断器 + 渐进恢复 |
| **无错误即错误** | 错误率为零反而意味着系统已不处理请求 | 同时告警"过多错误"和"过少请求" |
| **"客户端"/"预期"错误** | 将本应关注的错误归类为"不是我的问题" | 对客户端错误做异常检测 + 合成请求监控 |
| **版本化错误** | API 内外部表示耦合导致迭代困难 | 分离外部 API / 内部表示 / 存储层三个版本 |
| **"可选"组件的神话** | 缓存等"可选"组件悄然成为关键依赖 | 将所有组件视为必选组件进行运维 |
| **"清理了所有东西"** | 故障模式下的级联交互导致数据被误删 | 限流 + 熔断器 + 异常删除率检测 |
| **输入广度挑战** | 测试无法覆盖无限的输入空间 | 录制真实输入 + Fuzz 测试 |
| **处理过时工作** | 故障恢复后积压的过时请求拖慢系统 | 超时机制 + 基于延迟的自动扩缩 + 优先处理新请求 |
| **"第二系统"问题** | 推倒重来永远追不上老系统 | 渐进式微服务替换而非全量重写 |

### 深度解析几个关键模式

**惊群效应**可能是最致命的。想象一下：你的服务过载了，X 个请求超时。下一分钟，这 X 个请求重试 + X 个新请求 = 2X。再下一分钟就是 4X。指数增长可以在几分钟内彻底压垮系统。防御组合拳：**指数退避**（每次失败后加倍等待时间）+ **Jitter**（加入随机抖动，避免"集体冲锋"）+ **熔断器**（错误率超阈值时直接切断流量）+ **渐进恢复**（恢复时逐步放入流量而非瞬间 100%）。

**"清理了所有东西"**是书中最精彩的案例之一。一个照片存储系统有四个组件：用户数据库、REST API、文件存储、垃圾回收系统。正常流程：用户删号 -> 垃圾回收查询数据库确认用户不存在 -> 删除照片。但如果前端 API 在数据库不可用时返回 404 而不是 500？垃圾回收系统会认为**所有用户**都不存在，然后删除**所有照片**。这个案例告诉我们：分布式系统中，每个组件的故障都可能通过意想不到的路径引发级联灾难。系统设计必须考虑**组件交互在故障状态下的行为**，而非仅考虑正常状态。

**"第二系统"问题**是许多工程团队的切肤之痛。当现有系统积累了大量技术债务时，"从零重写"的诱惑极大。但现实是残酷的：旧系统在你重写期间还在不断演化，新系统"完成"时已经落后了六个月的功能。更糟的是新系统缺乏生产里程数——老系统的 bug 已经被发现和修复了，新系统的 bug 还潜伏着。正确的做法是利用分布式系统的天然优势：**逐个替换微服务**，而不是推倒整栋大厦。

## 总结

Part V 的三章内容为分布式系统实践提供了"软件之外"的视角：

1. **可观测性**不是锦上添花，而是系统可靠性的基础设施。日志、指标、告警、追踪四者缺一不可，且需要持续投资
2. **AI 集成**正在成为分布式系统的标配能力。关键是把它当作"又一个需要分布式思维来管理的组件"——关注延迟、成本、安全部署和测试范式转变
3. **故障模式**是最宝贵的知识——别人用停机时间和数据丢失换来的教训。记住：系统设计不仅要考虑一切正常时的行为，更要考虑**当某个环节出错时，其他环节会如何反应**

最后引用书中的一句话作为结尾——分布式系统的可靠性，最终由你的告警质量和故障恢复速度来定义。
