# 基础概念：从模式简史到分布式系统核心原语

> 本文是 Brendan Burns 所著 *Designing Distributed Systems, 2nd Edition* Part I（第 1-2 章）的阅读笔记。第 1 章讲 **为什么需要分布式模式**，第 2 章讲 **构建分布式系统前必须理解的核心概念**。

今天的应用——无论是消费级 App 还是支付后端——几乎都需要以分布式的方式来构建。然而分布式系统的设计和调试难度远高于单机程序。好消息是，容器化和容器编排（尤其是 Kubernetes）的普及，正在让分布式系统开发从 **黑魔法** 走向 **可学习、可复用的工程实践**。Part I 为后续章节打基础，帮我们建立一个关于分布式系统的共同语境。

## 第 1 章：软件模式的演进

### 模式简史：四次范式跃迁

软件行业中，"模式"（pattern）并不是第一次改变游戏规则。Burns 将模式的发展梳理为四个阶段：

| 阶段 | 时代 | 关键里程碑 | 核心贡献 |
|---|---|---|---|
| **算法形式化** | 1960s | Knuth《The Art of Computer Programming》 | 将算法从"特定机器的技巧"抽象为可学习、可复用的通用概念 |
| **面向对象模式** | 1990s | GoF《Design Patterns》 | 提供了接口级别的可复用模式（Factory、Observer、Adapter...），并配套语言特性落地为库 |
| **开源运动** | 2000s | Linux / 容器生态 | 让分布式基础设施本身（容器、编排器）成为社区共建产物 |
| **分布式系统模式** | 2010s- | 容器 + Kubernetes + 本书 | 将分布式系统的最佳实践提炼为可复用的蓝图（blueprint），而非特定技术栈的菜谱 |

> [!tip] 个人思考
> 每次范式跃迁都遵循相似的路径：**碎片化实践 -> 抽象与命名 -> 复用库/组件**。算法时代，Knuth 把排序和搜索从各家大型机的手册中抽象出来；OOP 时代，GoF 把接口模式从无数代码库中提炼出来。分布式时代，Burns 试图做同样的事——把 sidecar、ambassador、scatter/gather 等模式从各家微服务架构中抽象出来。

### 模式的三重价值

Burns 在书中明确阐述了模式对分布式系统开发者的三个核心价值：

**1. 站在巨人的肩膀上（Standing on the Shoulders of Giants）**

我们构建的系统可能是全新的，但它们遇到的工程问题——可靠性、可扩展性、敏捷性——并不新。模式让我们无需亲自踩坑就能获取他人的经验。就像学习算法不需要你自己先发明快速排序一样，学习分布式模式不需要你先亲历一次大规模宕机。

**2. 共享语言（A Shared Language for Discussing Our Practice）**

没有统一术语，团队沟通中会频繁出现 "violent agreement"——两个人在说同一件事，却因为各自用了不同名字而争论不休。当整个社区都认同 "sidecar" 的含义时，讨论可以直接跳到 **怎么用**，而不是 **那是什么**。

**3. 可复用组件（Shared Components for Easy Reuse）**

模式不仅是概念，还能落地为实际的可复用组件。例如，算法的形式化催生了排序库，OOP 模式催生了通用设计模式库，分布式模式则催生了以容器镜像形式存在的可复用构建块——它们通过 HTTP 接口实现跨语言复用。

> [!note] 关于软件供应链安全
> Burns 在第二版中特别提到：在软件供应链攻击频发的背景下，可复用组件的价值更加凸显。整个社区依赖同一个经过充分审计的核心实现，比每个团队各自造轮子要安全得多——依赖越少、审计越集中，风险面越小。

## 第 2 章：分布式系统核心概念

第 2 章是一份浓缩的分布式系统入门手册，覆盖了后续所有模式讨论都需要的基础原语。

### API 与 RPC

在分布式系统中，API 调用发生在网络上，通常以 **RPC（Remote Procedure Call）** 的形式存在。最常见的传输层是 HTTP + JSON（即 REST / RESTful），但也存在 gRPC、CORBA 等更结构化的协议。

API 可以分为两类：

| 类型 | 特征 | 适用场景 |
|---|---|---|
| **同步 API** | 请求完全处理后才返回 | 延迟在毫秒级的操作 |
| **异步 API** | 立即返回操作 ID，后续轮询状态 | 延迟在分钟级的长时操作 |

大型系统通常借助 **IDL（接口描述语言）** 如 OpenAPI 来自动生成客户端库，而不是让开发者手写 HTTP 调用。API 的核心功能通过 **SLO（Service Level Objective）** 来量化描述，涵盖延迟、可靠性和吞吐量。

### 延迟（Latency）

延迟是系统响应一次请求所需的时间，通常以毫秒为单位。测量延迟时有两个关键原则：

1. **确保代表真实用户体验**：测量的应该是端到端延迟，而非某个内部组件的处理时间。
2. **延迟必须可归因（attributable）**：你不仅要知道"系统慢了"，更要知道"慢在哪里"。

### 可靠性（Reliability）

可靠性 = 成功请求数 / 总请求数。但定义"成功"并不简单。以 HTTP 为例：

- **2xx**：明确成功
- **5xx**：明确失败（服务端错误）
- **4xx**：灰色地带——404 是客户端错误还是服务配置错误？403 是正常拒绝还是授权服务宕机？

> [!tip] 个人思考
> 实践中，把 404 一律算作客户端错误可能掩盖真正的可靠性问题——比如一次错误的路由配置导致大量 404。反过来，把所有 4xx 都算故障又会因为一个配置错误的爬虫而频繁误报。Burns 的建议是：正确的监控方式取决于具体应用，基于 AI 的异常检测在区分真实故障与噪声方面将越来越重要。

### 百分位数（Percentiles）

**不要用平均值来衡量分布式系统的性能**，这是 Burns 反复强调的一点。原因有二：

1. **算术平均被极值严重扭曲**：少量极低延迟请求会让平均值看起来比实际好很多。
2. **用户体验是多个请求的组合**：一个完整的用户操作可能触发 N 个请求，只要其中一个慢，整体体验就差。随着 N 增大，用户遇到至少一个慢请求的概率趋近 100%。

因此，P90/P99 百分位延迟（90% 或 99% 的请求在此延迟以下完成）才是更有意义的指标。

### 幂等性（Idempotency）

> 一个操作无论执行一次还是多次，结果都相同——这就是幂等性。

- 幂等操作：`X = 5`（无论执行几次，X 始终为 5）
- 非幂等操作：`X = X + 1`（每次执行结果都不同）

幂等性是分布式系统容错的基石：如果操作是幂等的，失败后直接重试即可，无需担心副作用。这与声明式配置（"我期望系统处于这个状态"）vs 命令式配置（"执行这些步骤"）的理念一脉相承。

### 交付语义（Delivery Semantics）

消息的交付语义分为三种：

<HtmlVisualization src="/book-notes/designing-distributed-systems/visualizations/delivery-semantics.html" height="520px" title="交付语义对比" />

| 语义 | 含义 | 典型场景 | 注意事项 |
|---|---|---|---|
| **At-least-once** | 消息至少送达一次，可能重复 | 日志收集、通知推送 | 需要消费端具备幂等能力 |
| **At-most-once** | 消息最多送达一次，可能丢失 | 信用卡扣款（宁可不扣也不能重复扣） | 失败时不重试 |
| **Exactly-once** | 消息恰好送达一次 | 理想情况 | 实践中极难实现，通常通过 at-least-once + 幂等来近似 |

> [!tip] 个人思考
> Burns 坦率地指出 exactly-once 在实践中几乎不可能真正实现。这一点在 Kafka 社区中引发过长期争论——Kafka 0.11 引入的 "exactly-once semantics" 实际上是通过事务性生产者 + 幂等写入来实现的，本质上仍然是 at-least-once + 去重。理解这一点对系统设计至关重要：**不要追求理论上的 exactly-once，而是设计好幂等机制**。

### 关系完整性（Relational Integrity）

当数据分散在多个存储系统中时（例如用户余额在一个服务，用户地址在另一个服务），如何保证它们之间的引用关系始终一致？这就是关系完整性问题。

- 传统 SQL 数据库通过事务和外键约束来保证强关系完整性
- NoSQL 数据库（如 Cassandra、CosmosDB）放松了关系语义，将一致性问题交给应用层处理

关系完整性的维护需要 **分布式事务** 和 **分布式锁**，代价是降低并行度和性能。这是一个性能与代码复杂度之间的权衡。

### 数据一致性（Data Consistency）

分布式系统为了容错和性能会将数据复制到多个数据中心，由此引出核心问题：**写入后数据多快在所有副本中变得一致？**

<HtmlVisualization src="/book-notes/designing-distributed-systems/visualizations/consistency-spectrum.html" height="600px" title="数据一致性频谱" />

**强一致性（Strong Consistency）**：
- 写入操作在所有副本确认后才返回成功
- 用户永远读到最新数据
- 代价：随数据中心数量增加，写入延迟和失败概率显著上升

**最终一致性（Eventual Consistency）**：
- 写入操作在单个数据中心确认后即返回，后台异步复制
- 系统更快、更可靠（不依赖所有数据中心同时可用）
- 代价：存在"读不到自己刚写的数据"（read-your-own-writes）问题

Burns 举了一个绝佳的例子：你在电商网站下了订单，然后立刻点"查看我的订单"——如果读请求被负载均衡到了还没同步的数据中心，你会看不到刚下的订单。**这种体验会让用户再也不在你这买东西。**

这里绕不开 **CAP 定理**：在一致性（Consistency）、可用性（Availability）和分区容忍性（Partition Tolerance）三者中，分布式系统只能同时满足两个。值得注意的是，CAP 并不是一个非此即彼的铁律——你可以在不同场景下做不同的权衡。比如 99% 的可用性对某些应用足够，那么就可以换取更强的一致性。

> [!warning] 选择一致性模型是架构级决策
> Burns 特别警告：一旦选择了强一致性或最终一致性，它会渗透到系统设计和应用代码的方方面面。切换存储模型的成本极高，决策的影响可能在数年后才完全显现。

### 编排与 Kubernetes

现代分布式系统依赖 **编排器（orchestrator）** 来管理部署和运行。编排器负责将声明式的期望状态（"我需要 3 个前端副本和 5 个后端副本"）映射到实际的机器资源上。目前最主流的编排器是 **Kubernetes**。

### 健康检查（Health Checks）

编排器需要知道应用的健康状态来做出调度决策。Kubernetes 区分了两种健康检查：

| 检查类型 | 含义 | 失败后果 |
|---|---|---|
| **Liveness Probe** | 应用是否还活着（进程是否响应） | 容器被终止并重启 |
| **Readiness Probe** | 应用是否准备好接收流量 | 从负载均衡器中移除，但不重启 |

一个刚启动的应用可能是 alive 但还不 ready——例如它正在下载缓存文件或预热模型权重。正确实现这两种探针，是分布式应用与编排器良好协作的基础。

```yaml
# Kubernetes Pod 健康检查配置示例
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: my-app
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 10
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 5
```

## 总结

Part I 的两章为全书建立了两个支柱：

1. **历史视角**：分布式系统模式不是凭空出现的，它是算法形式化 -> OOP 模式 -> 开源运动这条演进线的自然延伸。理解这个脉络有助于我们正确看待当下的技术实践。
2. **概念工具箱**：API/RPC、延迟、可靠性、百分位数、幂等性、交付语义、关系完整性、数据一致性、编排与健康检查——这些是后续讨论 sidecar、ambassador、adapter 等具体模式的共同语言。

> [!tip] 一句话带走
> 分布式系统设计的核心挑战不在于实现某个功能，而在于 **在不可靠的基础设施上，以可接受的延迟，向用户提供一致且可靠的服务**。Part I 给出的这些概念，正是用来精确描述和讨论这个挑战的工具。
