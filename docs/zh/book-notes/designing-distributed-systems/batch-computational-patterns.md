# 批处理计算模式：工作队列、事件驱动与协调批处理

> 本文是 Brendan Burns《Designing Distributed Systems, 2nd Edition》Part IV（Chapter 11-13）的阅读笔记。这三章覆盖了分布式批处理的三种核心编排模式：工作队列、事件驱动批处理和协调批处理。

前面章节讨论的都是长期运行的服务端应用模式。Part IV 转向 **批处理**（batch processing）——这类任务只运行一段有限的时间，处理完就结束。典型场景包括：用户遥测数据聚合、销售数据日报/周报、视频转码等。批处理的核心诉求是 **用并行加速大规模数据处理**，而最著名的分布式批处理模式就是 MapReduce。但除了 MapReduce，还有很多实用的批处理模式值得深入理解。

## Chapter 11：工作队列系统（Work Queue Systems）

### 基本架构

工作队列是最简单的批处理形式。核心思想：有一批彼此独立的工作项（work items），每个都可以单独处理，互不影响。系统需要确保每个工作项在规定时间内被处理完毕，通过动态伸缩 Worker 来应对负载变化。

| 组件 | 职责 | 实现方式 |
|------|------|----------|
| **Work Queue Manager** | 追踪工作项状态，协调分配 | 通用库容器（可复用） |
| **Source Container** | 提供工作项列表 | Ambassador 模式，应用特定 |
| **Worker Container** | 实际执行处理逻辑 | 独立容器组，应用特定 |

### 容器化工作队列的接口设计

书中提出了一个很巧妙的分离原则：将 **通用队列管理逻辑** 与 **应用特定逻辑** 分开，通过两个接口实现模块化。

**Source Container 接口**（REST API）：

```
GET /api/v1/items          → 返回所有工作项列表
GET /api/v1/items/<name>   → 返回特定工作项详情
```

Source Container 本质上是一个 Ambassador，将外部数据源（云存储、Kafka 队列、NFS 文件列表等）适配为统一的 REST 接口。

**Worker Container 接口**（基于文件）：

Worker 启动时通过环境变量 `WORK_ITEM_FILE` 获取一个本地文件路径，该文件包含工作项数据（通过 Kubernetes ConfigMap 挂载）。这种基于文件的接口更安全——只需一次调用，避免了外部恶意注入工作的风险。

::: tip 为什么 Source 用 REST、Worker 用文件？
Source Container 和 Manager 在同一个容器组内，需要反复通信，REST 最方便。Worker 是独立容器组，只需在启动时获取一次工作数据，文件接口更简单也更安全。这种"因地制宜"的接口选择体现了实用的工程思维。
:::

### 共享基础设施的实现

Work Queue Manager 的核心算法：

1. 从 Source Container 加载可用工作项
2. 查询已创建的 Kubernetes Job，确定哪些工作项已完成或正在处理
3. 对未处理的工作项，创建新的 Job 对象来启动 Worker Container
4. 循环往复

Kubernetes 的 Job 对象天然支持可靠执行——即使某台机器宕机，Job 也会在其他节点重新调度。这意味着 Work Queue Manager 不需要自己实现复杂的故障恢复逻辑，大大简化了开发。

### 实战：视频缩略图生成器

书中给了一个具体案例：为视频生成缩略图。

- **Source Container**：一个 Node.js 程序，扫描 NFS 共享目录中的 `.mp4` 文件，返回文件列表
- **Worker Container**：使用 `ffmpeg -i $INPUT_FILE -frames:v 100 thumb.png` 抽取每 100 帧一张缩略图

通过定义简单的 Source 和 Worker 容器，就能快速搭建出一个完整的容器化队列处理系统。从"有一个需求"到"能跑的实现"，距离被大幅缩短。

### 动态伸缩 Workers

静态的工作队列在面对突发负载时会遇到问题：要么过度预留资源浪费成本，要么资源不足导致延迟不断增长。书中提出了一个基于两个关键指标的自适应伸缩方案：

| 指标 | 定义 | 用途 |
|------|------|------|
| **到达间隔时间**（interarrival time） | 工作项数量 / 时间跨度 | 衡量输入速率 |
| **处理延迟**（latency） | 开始处理到完成的时间 | 衡量处理能力 |

**核心决策逻辑**：如果 `处理延迟 > 到达间隔时间`，队列会不断积压，必须增加并行度。并行处理时，有效处理时间 = 单项处理时间 / 并行数。例如单项处理需 1 分钟，4 个并行 Worker 的有效处理时间就是 15 秒，可以承受 16 秒以上的到达间隔。

缩容时则需要更保守——作者建议保持处理时间不超过到达间隔的 90%，留出安全余量。实际生产中，可以使用 [KEDA (Kubernetes Event-Driven Autoscaling)](https://keda.sh/) 来实现基于事件源的自动伸缩。

### Multiworker 模式

当一个工作项需要多种处理（比如人脸检测 → 身份识别 → 模糊处理）时，可以将多个独立的 Worker 容器组合成一个 **Multiworker 聚合容器**。这本质上是 Adapter 模式的特化——一个统一接口容器在内部委派给多个具体处理容器。

好处是每个处理步骤都是独立可复用的。比如"模糊处理"容器既能用于人脸模糊，也能用于车牌模糊，不同的工作队列可以共享这些组件。

## Chapter 12：事件驱动批处理（Event-Driven Batch Processing）

### 从工作队列到工作流

当单个工作队列的 **输出成为另一个队列的输入** 时，简单的工作队列就演变为 **工作流**（workflow）——一个有向无环图（DAG），描述了多阶段处理的流转关系。

事件驱动批处理的运作类似于 FaaS（Function as a Service），但没有一个整体蓝图来描述各个队列之间的关系，这使得理解系统全貌变得困难。而且，虽然初始设计可能简单，正确处理错误和意外情况会让系统迅速变得复杂。

### 五种核心模式

#### 1. Copier（复制器）

将一个输入流 **复制** 到两个或多个相同的输出流。

**典型场景**：视频转码——同一个原始视频需要同时生成 4K、1080p、移动端低分辨率和 GIF 缩略图四种格式，每种格式由一个独立的工作队列处理。

**隐藏妙用**：通过 Copier 将同一个工作项交给两个相同的 Worker 并行处理，取先完成的结果，可以 **对冲尾延迟**（tail latency）。如果 Worker 有 1% 的概率出现 60 倍慢速，双路处理可以将可靠性从 99% 提升到 99.99%。

#### 2. Filter（过滤器）

按条件 **丢弃** 不符合要求的工作项，输出的是输入的子集。

**典型场景**：新用户注册流程中，过滤掉未勾选"接收营销邮件"的用户。Filter 作为 Ambassador 包裹原始 Source Container，只返回符合条件的工作项。

#### 3. Splitter（分流器）

类似 Filter，但 **不丢弃**任何工作项——而是按条件将输入分发到不同的输出队列。

**典型场景**：处理物流发货通知——Splitter 将"偏好短信通知"的订单路由到短信队列，将"偏好邮件通知"的订单路由到邮件队列。一个工作项也可以同时进入多个队列（此时 Splitter 同时扮演了 Copier 的角色）。

> 有趣的是，Splitter 可以用一个 Copier + 两个 Filter 实现。这就像 XOR 门可以用 OR/NOT/AND 门组合实现一样——Splitter 只是一个更紧凑的表达。

#### 4. Sharder（分片器）

Splitter 的泛化形式。Sharder 基于 **分片函数**（sharding function）将输入均匀分配到多个队列。

**两大好处**：

- **故障隔离**：如果某个 Worker 镜像更新导致崩溃，只有对应分片受影响。4 个分片意味着最多只有 1/4 的用户受波及，支持分阶段灰度发布
- **负载分散**：将工作均匀分布到多个数据中心/区域，同时提供跨区域容灾能力

当某个分片的队列不可用时，Sharder 会 **动态调整**，将所有流量导向剩余健康的队列。

#### 5. Merger（合并器）

Copier 的反向操作——将 **多个输入队列合并** 为一个输出队列。

**典型场景**：多个代码仓库同时触发构建请求，每个仓库作为一个独立的 Source，Merger 将所有构建请求合并到统一的构建系统。这是 Adapter 模式的又一个应用——多个 Source Container 通过 Merger Adapter 适配为单一 Source。

### 五种模式总结

| 模式 | 输入 | 输出 | 语义 |
|------|------|------|------|
| **Copier** | 1 个队列 | N 个相同队列 | 一对多复制 |
| **Filter** | 1 个队列 | 1 个队列（子集） | 条件丢弃 |
| **Splitter** | 1 个队列 | N 个不同队列 | 条件路由 |
| **Sharder** | 1 个队列 | N 个均匀队列 | 哈希分片 |
| **Merger** | N 个队列 | 1 个队列 | 多对一合并 |

这五种模式是 **可组合的构建块**，可以像搭积木一样组合出任意复杂的处理流水线。

### Pub/Sub 基础设施：Kafka

当工作流变得复杂时，需要一个可靠的消息传递基础设施来连接各个阶段。书中选择了 **Apache Kafka** 作为 Pub/Sub 实现。

Kafka 的核心概念：

- **Topic**：消息的逻辑通道。工作流中每个阶段的输出通常对应一个 Topic
- **Partition**：每个 Topic 可以划分为多个分区，实现负载均衡。例如一个 Sharder 输出 3 个分片，就创建 `Photos-1`、`Photos-2`、`Photos-3` 三个 Topic
- **Replication Factor**：每条消息复制到多少台机器，建议 3 或 5 以保证容错

在 Kubernetes 上可以通过 Helm 快速部署 Kafka：

```bash
helm repo add incubator http://storage.googleapis.com/kubernetes-charts-incubator
helm install --name kafka-service incubator/kafka
```

### 弹性与性能

#### Work Stealing（工作窃取）

现实中，不同工作项的处理时间差异很大。当所有"慢任务"恰好集中在同一个分片时，该分片就会变成瓶颈——就像超市里最慢的收银台。

**Work Stealing 算法**：当一个 Worker 的队列为空时，不是空等，而是 **从最长的队列尾部取走工作项**。从尾部取而非头部，是为了避免和原队列的 Worker 产生竞争。这样所有队列的长度都趋向最小值，整体延迟更均匀。

#### 错误处理与重试

Worker 崩溃是常态。消息队列通常采用 **隐藏式取出** 语义：消息被取出后并非真正删除，而是对其他 Worker 不可见；处理完成后发送 ACK 才真正删除。超时未 ACK，消息重新变为可见。

这引出几个关键问题：

| 问题 | 解决方案 |
|------|----------|
| 两个 Worker 处理同一项 | 确保工作项 **幂等**（idempotent） |
| "毒药"工作项反复崩溃 Worker | 追踪重试次数，**指数退避**（exponential backoff） |
| 下游故障导致大量积压 | **双队列策略**：新消息队列 + 重试队列，新消息优先处理 |

::: warning 积压风暴
当下游依赖故障恢复后，积压的旧工作项会和新工作项竞争处理资源。如果旧工作项已经过时且新工作项有时效性，双队列策略至关重要——它确保新消息不会被排在重试消息后面。这本质上是一个 **优先级队列** 的应用。
:::

### 实战：新用户注册工作流

书中用一个完整的用户注册流程演示了模式组合：

1. **Sharder** → 将新注册用户分片到多个验证邮件队列（跨地理区域，容灾）
2. 用户验证邮件返回 → 进入新的工作流
3. **Copier** → 复制到两个队列：欢迎邮件队列 + 通知设置队列
4. 通知设置队列经过 **Filter** → 分为邮件通知队列和短信通知队列

一个看似简单的"用户注册"，被分解为 Sharder → Copier → Filter 的组合，每个环节独立可伸缩、可监控、可替换。

## Chapter 13：协调批处理（Coordinated Batch Processing）

### 问题：需要"等所有人到齐"

前两章的模式都是 **发散型** 的——将工作拆分、复制、分流。但有些场景需要 **汇聚**：等所有并行处理完成后，再进行下一阶段（聚合统计、生成最终结果等）。这就是协调批处理要解决的问题。

### Join（Barrier 同步）

Join 类似于线程编程中的 `thread.join()`——所有并行工作队列的输出都到齐后，才释放到下一阶段。这也叫 **Barrier 同步**。

**典型场景**：动画渲染。每一帧可以独立渲染，但组合为 1 秒视频时必须等 30 帧全部完成。

**Join 的价值**：确保聚合操作（如求和、计数）时数据集是完整的。

**Join 的代价**：必须等最慢的分片完成，限制了整体并行度，增加了端到端延迟。可以通过 Copier 双路处理每个工作项来对冲尾延迟。

### Reduce（归约）

如果 Sharding 是 MapReduce 中的 **Map**，那么 Reduce 就是另一半。Reduce 将多个并行输出合并为单一的聚合结果。

与 Join 的关键区别：Reduce **不需要等所有数据到齐** 就能开始工作。它可以乐观地（optimistically）合并已有的输出，然后反复归约，直到只剩一个最终结果。

这是一个巨大的优势——Reduce 可以在 Map 阶段还在运行时就开始执行，整体执行时间更短。

#### 三种 Reduce 操作

**1. Count（计数）**

统计某事物出现的次数。例如统计一本书中每个单词出现的次数：

1. 将书按页码分片到 10 个队列
2. 每个 Worker 统计自己负责的页中的词频
3. Reduce 阶段将多个词频表合并（同一个词的计数相加）

每个 Worker 的输出如 `{a: 50, the: 17, cat: 2}`，两个输出归约后变成 `{a: 80, the: 42, cat: 2}`。这个过程可以递归执行直到只剩一个结果。

**2. Sum（求和）**

类似 Count，但累加的是实际数值。例如统计美国总人口：

1. 第一层分片：按州分配工作队列
2. 第二层分片：每个州内按县分配
3. 每个 Worker 统计其负责的城镇人口
4. Reduce 阶段逐层合并：`(Seattle, 4,000,000)` + `(Northampton, 25,000)` → `(Seattle-Northampton, 4,025,000)`

多层分片 + 并行 Reduce，大部分计算都在并行执行。

**3. Histogram（直方图）**

在求和的基础上增加分布统计。例如在统计人口的同时，构建美国家庭子女数量的分布模型：

```
0个子女: 15%
1个子女: 25%
2个子女: 45%
3个子女: 10%
4个子女:  5%
```

合并直方图时，需要用各自的总人口作为权重：将每个直方图的百分比乘以其对应的总人口，得到绝对数量后相加，再除以合并后的总人口得到新的百分比。这比简单的计数或求和更精妙，但同样满足 Reduce 的可递归合并性质。

### 实战：图像标注处理流水线

书中最复杂的综合案例——高速公路监控图像的批处理分析：

**目标**：统计高峰时段的车辆数量（轿车/卡车/摩托）及颜色分布，同时保护隐私（模糊车牌）。

**完整流水线**：

```
原始图像队列
    │
    ▼
  Sharder ──────────────────────────── 分片到多个处理队列
    │                    │
    ▼                    ▼
 模糊队列 #1         模糊队列 #2 ────── Multiworker: 车牌检测 + 模糊处理
    │                    │
    └────────┬───────────┘
             ▼
           Join ─────────────────────── 等所有分片完成
             │
             ▼
           Copier ───────────────────── 复制到两个队列
           ╱    ╲
          ▼      ▼
     删除原图  图像分析队列
     队列        │
                 ▼
               Sharder ──────────────── 再次分片
               │          │
               ▼          ▼
           分析队列 #1  分析队列 #2 ── Multiworker: 车辆识别 + 颜色检测
               │          │
               └────┬─────┘
                    ▼
                  Reduce ────────────── 归约统计结果
                    │
                    ▼
              最终统计报告
```

这个流水线综合运用了前两章的几乎所有模式：

| 阶段 | 使用的模式 | 目的 |
|------|-----------|------|
| 图像分片 | Sharder | 并行化 + 故障隔离 |
| 车牌模糊 | Multiworker | 组合复用（检测 + 模糊） |
| 等待所有模糊完成 | Join | 确保删除原图前所有副本已处理 |
| 分发到删除和分析 | Copier | 并行执行两个独立任务 |
| 车辆分析 | Sharder + Multiworker | 并行化 + 组合复用（识别 + 颜色） |
| 汇总统计 | Reduce | 合并所有分片的统计结果 |

::: info 为什么在 Copier 之前要 Join？
因为删除原图是不可逆操作。必须确保 **所有** 分片都完成了模糊处理后才能安全删除原图，否则在灾难恢复场景下可能丢失未处理的图像。Join 在这里提供的不是性能优化，而是 **正确性保证**。
:::

## 全景总结：三层批处理模式

| 层次 | 模式 | 数据流方向 | 核心价值 |
|------|------|-----------|----------|
| **基础** | Work Queue | 一对多（并行处理） | 独立任务的并行化 |
| **编排** | Copier / Filter / Splitter / Sharder / Merger | 多对多（流转路由） | 复杂工作流的模块化组合 |
| **协调** | Join / Reduce (Count, Sum, Histogram) | 多对一（汇聚聚合） | 分布式结果的同步与归约 |

这三层模式从下到上，构成了一个完整的批处理系统设计工具箱。每个模式都是一个独立的、可复用的构建块，通过组合就能表达任意复杂的批处理流水线。

Brendan Burns 在这部分的设计哲学非常清晰：**用容器化和标准接口把通用逻辑与业务逻辑分离**。Work Queue Manager、Source Container 接口、Worker Container 接口——这些都是可以做成通用库的基础设施，开发者只需实现自己的 Source 和 Worker 就能快速搭建批处理系统。这与 Unix 哲学"组合简单工具完成复杂任务"一脉相承，只不过构建块从命令行程序变成了容器。
